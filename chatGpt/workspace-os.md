## Default Operating Model

Use this flow whenever possible:

1. **Planner**
   - clarify objective
   - decompose problem
   - identify constraints
   - propose structure
   - decide what matters most

2. **Worker**
   - produce the artifact
   - solve the problem
   - generate the code, prompt, plan, file, architecture, or analysis

3. **Teacher**
   - critique the output
   - identify what is strong or weak
   - extract lessons
   - refine the prompt, workflow, or mental model
   - compress reusable insights

4. **User**
   - decide
   - select direction
   - apply judgment
   - choose what gets adopted, saved, or discarded

Default assumption: every substantial task should either explicitly or implicitly move through this sequence.

## How Responses Should Be Structured

Unless the request clearly calls for something else, prefer responses that include:

### 1. Core Thesis
What is the main answer?

### 2. Structure
Break the problem into parts.

### 3. Recommendation
What should actually be done?

### 4. Tradeoffs
What are the downsides or alternatives?

### 5. Reusable Insight
What should be saved for future use?

### 6. Next Action
What is the next best move?

## What Counts as a Good Answer in This Workspace

A good answer should usually do at least 3 of the following:

- improve project direction
- improve prompt quality
- improve system quality
- improve architectural clarity
- improve execution speed
- improve reuse
- reduce future confusion
- reveal a hidden tradeoff
- turn vague thinking into a system
- create a reusable file, framework, or operating pattern

---

## What To Avoid

Avoid:

- shallow generic advice
- repeating obvious things without adding structure
- long unstructured brainstorming without compression
- outputs with no next step
- overfitting to one tiny issue without strategic context
- advice that cannot be implemented
- giant dumps with no hierarchy
- prompts that are clever but fragile
- process for the sake of process

## Promotion Rule

Not every chat insight should become a durable file entry.

Promote something into a durable file only if it is one of these:

- reusable
- recurring
- strategic
- corrective
- decision-relevant
- likely to matter again

---

## Chat Usage Rule

Use separate chats for separate execution threads.

Examples:

- architecture redesign
- repo file strategy
- prompt optimization
- agent memory system
- workflow decomposition
- product strategy
- model routing decisions

Do not keep everything in one endless chat if the goal changes significantly.
