# Screen Review Format

Use this format for screenshots, local app reviews, screen specs, or product docs that describe a screen.

## Screen: {name}

**Purpose**
What user goal does this screen serve?

**First 3 seconds**
What does the user understand immediately? What is missing, noisy, or misleading?

**Primary action**
What is the main action? Is it visually and semantically dominant?

**Hierarchy**
What does the eye see first, second, third? Does that match the task?

**Interaction model**
What can the user directly manipulate? What is indirect, detached, or hidden?

**Friction**
Where might the user hesitate, lose trust, or need to remember something?

**Feedback and state**
Are loading, empty, success, error, disabled, in-progress, and result states designed?

**Learning loop**
How does the screen support attempt, correction, retry, reuse, and progress?

**Premium feel**
What feels calm, fast, intentional, trustworthy, and delightful? What feels cheap or generic?

**Benchmark calibration (optional)**
For high-impact screens or redesign options, which 5-10 relevant apps define the bar for this pattern? How does this screen rank against them on clarity, interaction directness, emotional safety, state polish, premium feel, and time to first value? Where should your product match conventions or intentionally diverge?

**Accessibility**
Keyboard, screen reader, contrast, readable size, touch target, focus, and reduced-motion concerns.

## Flow Review Additions

For multi-screen flows, also capture:

- Entry point
- Exit point
- Time to first value
- Activation moment
- Trust breaks
- State carried between screens
- Places where the user can recover or retry
- Places where the product should remember context
